# NanoPol カメラ制御および測定・保存仕様

本仕様書は、ナノ粒子や量子ドットからの散乱光強度を正確に測定するための、カメラハードウェアの制御、データ保存フォーマット、およびストリーミング配信のアーキテクチャについて定義する。

## 1. 基本方針 (設計思想)
*   **汎用性の完全確保:**
    測定に使用する波長（785nm, 637nm等）やサンプルに依存せず、常に正確な「相対強度」を測定・規格化可能にする。
*   **ブラックボックス処理の排除:**
    人間が見やすい画像を作るためのソフトウェア的な自動色補正（ThorCam等が裏で行っているガンマ補正やホワイトバランスの自動調整）を完全に排除し、カメラを「純粋な光子カウンター」として運用する。

## 2. モード別の測定・保存仕様
システムは、ユーザーの目的に応じて「Colorモード」と「Monochrome (RAW) モード」の2つのカメラ動作モードを提供する。

### 2.1. Color モード (人間向けの確認・記録用)
*   **用途:** サンプルの色味を確認したい場合や、手軽な記録。
*   **状態別の動作:**
    *   **待機中 (プレビュー):** カメラによるデモザイク処理・補正を経たフルカラー映像で配信（最大 30fps）。
    *   **Snapshot (1枚撮影):** プレビューと同じ 8-bit カラー画像（JPEG / PNG / TIFF 等）として保存する。
    *   **録画:** `8-bit Video` (MP4 または AVI) 形式に固定。プレビューは30fpsのまま記録される。

### 2.2. Monochrome / RAW モード (解析・精密測定用)
光強度の定量評価（Snapshot / Step & Shoot / 16-bit録画）を行うための本番用モード。

1.  **データ取得の基本原則:**
    カメラセンサーから `10-bit` または `16-bit RAW` データ（Bayer配列）を直接取得し、ソフトウェアによるデモザイク（補間処理）を一切行わない。
2.  **ビニング処理 (Binning):**
    多様な波長（特にRGBフィルターの境界が曖昧になる近赤外領域など）に対応するため、取得したRAWデータの **「2x2の4画素（R, G, G, B）」の数値を単純加算（ソフトウェアビニング）** し、これを1つの大きなピクセルの光強度データとする。
3.  **保存形式:**
    RGBチャネルを持たない **「1チャンネルの 16-bit TIFF」**（録画時はマルチページTIFFまたは連番TIFF）形式で保存する。これによりファイルサイズを抑えつつ、その後の解析（NumPy等）を容易にする。
4.  **ハードウェア設定の完全固定:**
    測定の比例関係（リニアリティ）を崩さないため、バックエンド側で以下の設定を強制する。
    *   `Gamma`: **1.0 に固定**（無効化）。
    *   `White Balance (AWB)`: **Autoをオフ**にし、RGBのゲインをすべて **1.0 に固定**。
    *   `Hardware Gain`: 手動で指定された値に固定。

5.  **状態別のシステム挙動:**
    *   **待機中 (プレビューのみ):**
        PCへの負荷を下げるため、カメラは「8-bit モノクロモード」で動作させ、滑らかな白黒映像（最大 30fps）を配信し続ける。
    *   **Snapshot (1枚撮影) の瞬間:**
        アプローチA（モード切替）を採用。撮影ボタンが押された瞬間、カメラを「16-bit RAWモード」に切り替えて純粋な強度データを取得・ビニングし、TIFF保存する。その後すぐ待機中の8-bitモードに戻すため、**撮影の瞬間はプレビューが一瞬停止する**。
    *   **動画録画中:** 選択された録画モードにより動作が分岐する。
        *   **モード A [ 8-bit Video ]:**
            用途: ざっくりとした光の変化を後で見返すための軽量な記録。動作: 待機中と同じ8-bitモノクロモードのまま、プレビュー映像を動画ファイルとして保存する。ストリーミングは滑らかに継続する。
        *   **モード B [ 16-bit RAW (High Precision) ] 【本番測定用】:**
            用途: 画像解析を用いた正確な光強度の数値化・グラフ化。動作: 録画開始時にカメラを「16-bit RAWモード」に切り替え、全力で取得・ビニングし、TIFFとしてディスクに保存し続ける。この間、ストリーミングへの分配は負荷を下げるため、RAWデータを元に白黒映像を生成し **「1秒間に1〜2回 (1~2 fps)」に極端に間引いて** 配信する。録画終了後、待機中のモードに戻る。

## 3. バックエンド・アーキテクチャ (負荷分散とデータ欠損防止)
「16-bit TIFFの全力保存」と「ストリーミング用のJPEG圧縮」を同時に行うとCPU負荷が跳ね上がり、測定データがコマ落ち（欠損）する致命的な問題が発生する。これを防ぐため、バックエンド（Python）では **「Drop-oldest キューを用いたマルチスレッド非同期設計」** を採用する。

1.  **保存用スレッド (特急レーン):**
    *   役割: カメラからのRAWデータ取得とTIFFへの保存「のみ」を最優先で実行する。重い変換処理は一切行わない。
    *   連携: 保存が完了した画像のコピー（参照）を、ストリーミング用スレッドの共有領域（机の上）に配置する。カメラから次の画像が来た場合、古い画像は容赦なく上書きされる（Drop-oldest）。
2.  **ストリーミング用スレッド (各駅停車レーン):**
    *   役割: 自身のペース（例: 0.5秒に1回）で、共有領域にある「その時の最新の画像」を手に取る。
    *   処理: 時間をかけて 8-bit への変換、白黒化、JPEG圧縮を行い、ブラウザへ送信する。
    *   効果: このスレッドの処理がどれだけ遅延しても、保存用スレッドのデータ受信・保存には 1ミリ秒たりとも影響を与えない。

この設計により、**「測定データの完全性の保証」と「プレビュー映像のリアルタイム性の維持」を両立**させる。

## 4. メモリ管理とスレッド間通信 (実装方針)
マルチスレッド環境下におけるメモリアクセス違反を防ぎ、16-bit RAWデータの欠損（コマ落ち）を完全に排除するため、バックエンド（Python）では以下のメモリ管理ポリシーを厳格に適用する。専門用語が多いため、それぞれの目的と「現実世界の作業」に例えた補足を付記する。

### 4.1. ハードウェアバッファの動的割り当て
カメラがPCのRAM上に直接データを書き込むための「画像バッファ（予約席）」は、現在の状態（Idle, Snapshot, Rec）に応じて動的に再割り当てを行う。
*   **待機中 (8-bit) / 録画モードA:** 1画素あたり 1バイト (8-bit) のメモリ空間（小さな箱）を確保。
*   **Snapshot / 録画モードB:** 1画素あたり 2バイト (16-bit) のメモリ空間（大きな箱）を確保。
*   **【補足説明】:** カメラに「これから送るデータのサイズが変わるから、それに合わせた新しいキャンバス（メモリ）を用意してね」と指示する処理です。撮影ボタンを押した瞬間などに、古いキャンバスを片付けて新しいキャンバスを準備します。

### 4.2. ディープコピーによるメモリ隔離 (画像破綻の防止)
カメラ（uEye SDK）は、確保したバッファに対してハードウェアレベルで絶えず最新の画像データを上書きし続ける。ストリーミング用スレッドが処理を行っている最中にデータが上書きされて画像が破綻（ティアリング）するのを防ぐため、**「メモリの完全な複製 (Deep Copy)」** を徹底する。
*   **【補足説明】:** カメラは「超高速で同じキャンバスに何度も上書きし続ける画伯」です。もし私たちがそのキャンバスを直接覗き込みながら時間をかけて配信の準備をしていると、途中で画伯が新しい絵を上書きしてしまい、「上半分が1秒前の絵、下半分が今の絵」というようなバケモノ画像が生まれます。これを防ぐため、絵が完成した瞬間に「サッとコピー機で複製（ディープコピー）」を作り、私たちはそのコピー用紙を別の部屋に持っていって配信作業を行います。

### 4.3. スレッド間共有キュー (Drop-oldestパターンの実現)
保存用スレッド（特急レーン）からストリーミング用スレッド（各駅停車レーン）へコピーした画像を渡す際は、**「最大サイズ 1」に制限されたスレッドセーフな共有キュー** を使用する。
*   **挙動 (Non-blocking):** 保存用スレッドは、キューへアクセスする際に **「1マイクロ秒も待機しない (Non-blocking)」**ことを絶対条件とする。キューに古い画像が残っていれば即座に破棄して新しい画像を配置する。また、ストリーミングスレッドがキューにアクセス中でロック（鍵）がかかっていた場合、保存用スレッドは画像の配置自体を**「即座に放棄 (諦める)」**して次のカメラ受信へ戻る。
*   **【補足説明 - なぜ待たないのか？】:** 「キュー」とは、スレッド間でデータを手渡すための「鍵付きトレイ」のことです。「最大サイズ1」にすることで、古い画像が自動的に破棄されてメモリのパンクを防ぎ、配信スタッフは常に「最新の画像」を受け取れます。さらに、保存スタッフは「トレイが塞がっていたら待たずに諦める」というルールにすることで、配信側の遅延が保存側のデータ欠損（コマ落ち）を引き起こすリスクを理論上「完全にゼロ」にすることができます。

### 4.4. ガベージコレクション (GC) 負荷への備え
毎秒何十回も数MBのディープコピーを作成・破棄するため、Pythonの自動メモリ掃除機能（ガベージコレクション）による瞬間的な処理停止（マイクロスタッター）が発生するリスクがある。初期実装ではPython標準のメモリ管理に任せるが、コマ落ちが確認された場合は **「メモリプーリング」** への移行を検討する。
*   **【補足説明 - メモリプーリングとは？】:** コピー用紙を毎秒30枚も印刷してはゴミ箱に捨てていると、Pythonの清掃員（ガベージコレクション）が「ちょっと作業止めるよ！ゴミ捨てるから！」と割り込んできます。この一瞬の停止がコマ落ちの原因になります。これを防ぐ究極の対策が「メモリプーリング」です。これは紙を使うのをやめて「3枚のホワイトボードを準備し、それを消して使い回す」という手法です。ゴミが出なくなるため、清掃員が介入してくることがなくなります。まずは通常通り（毎回新しい紙を作り、古い紙は自動消滅させる方式）で作りますが、もしコマ落ちするようならこのホワイトボード方式を導入します。

### 4.5. Pythonのメモリ管理と参照カウントの活用 (技術的背景)
システムが際限なくメモリを消費（メモリリーク）せず、かつ超高速に動作する裏付けとして、Pythonの「参照」の仕組みを最大限に活用する。
*   **キューの中身は「実体」ではなく「参照」:** スレッド間で受け渡しされるのは、数MBの画像データそのものではなく、メモリ上の位置を示す「参照（名札・ポインタのようなもの）」のみである。そのため、キューへの出し入れは数マイクロ秒で完了する。
*   **取り出し (Pop) による完全消費:** ストリーミングスレッドがキューから画像（参照）を読み取る動作は、単なる値のコピーではなく「キューからの引き抜き (Dequeue)」である。取り出した直後にキューは完全に「空」になるため、参照が意図せず増殖したり、古い参照がトレイに残り続けることはシステム構造上あり得ない。
*   **Drop-oldestと自動消滅:** 保存用スレッドが新しい画像の参照をキューに入れ、古い参照を押し出した（Dropした）際、もしストリーミングスレッドがその古い参照を持っていなければ、該当画像への「参照カウント」が0になる。Pythonはこの状態を検知し、即座にメモリから実体データを完全に解放する。
*   **結果:** 毎秒30回新しい画像メモリ（約2.6MB）を確保し続けても、メモリ上に同時に存在するのは「キューの中の最新の1枚」と「ストリーミング処理中の1枚」の最大2枚（約5.2MB）のみに保たれ、システムのメモリパンクを完全に防ぐことができる。
