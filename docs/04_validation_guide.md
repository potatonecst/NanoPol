# 04. 入力バリデーションと数値処理 (Validation Guide)

このドキュメントでは、nanopol プロジェクトで使用されている入力値のチェック（バリデーション）と、特に注意が必要な「小数の扱い」について解説します。

## 1. なぜバリデーションが必要なのか？

ハードウェア（モーターやレーザーなど）を制御する際、不適切な値を送ると以下のような問題が起こる可能性があります。

1.  **機器の故障**: 許容範囲を超えたパワーや速度を指定してしまう。
2.  **動作不良**: モーターが物理的に動けない角度（ステップの途中など）を指定してしまう。
3.  **予期せぬ挙動**: プログラムが想定していないデータ形式（文字など）が混入する。

これらを防ぐため、ユーザーが入力した瞬間に「その値が正しいか」をチェックするのがバリデーションの役割です。

## 2. 「0.0025」という数字の秘密

手動制御画面（Manual Control）では、角度入力に対して **`0.0025` の倍数** であることを求めています。

```typescript
const STEP_RESOLUTION = 0.0025;
```

### なぜ 0.0025 なのか？
これは使用しているモーター（GSC-01 / OSMS-60YAW）の「最小ステップ角」や、ギア比などのハードウェア仕様に由来します。
このモーターは無限に滑らかに動くわけではなく、カクッカクッと決まった幅でしか止まれません。その最小単位が `0.0025` 度なのです。

例えば、`0.0026` 度に動かそうとしても、モーターは物理的にその位置で止まることができません。そのため、プログラム側で入力を拒否するようにしています。

## 3. コンピュータは小数が苦手？（浮動小数点問題）

バリデーションのコードには、少し不思議な計算が含まれています。

```typescript
// src/schemas/manualControlSchema.ts より抜粋
const isMultipleOfStep = (val: number) => {
    // ...
    const divided = val / STEP_RESOLUTION;
    const rounded = Math.round(divided);
    return Math.abs(divided - rounded) < EPSILON;
};
```

単純に `val % 0.0025 === 0` （割り切れるか？）と書かないのはなぜでしょうか？

### 0.1 + 0.2 は 0.3 ではない
コンピュータの世界（IEEE 754 規格）では、小数を2進数で表現するため、人間に馴染みのある10進数の小数を正確に表現できないことがあります。

- 人間の計算: `0.0050 / 0.0025 = 2` (割り切れる)
- コンピュータ: `0.0050 / 0.0025` が `1.99999999999999` のようになることがある

この微小なズレを「丸め誤差」と呼びます。

### 解決策：EPSILON（イプシロン）
誤差を許容するために、「計算結果が整数に **限りなく近ければ** OK」という判定を行っています。

1.  **`divided`**: 入力値をステップ幅で割る（理想的には整数になるはず）。
2.  **`rounded`**: それを四捨五入して整数にする。
3.  **`Math.abs(divided - rounded) < EPSILON`**:
    - 実際の値と、整数の差が `0.001` (EPSILON) より小さければ、「これは誤差の範囲内であり、実質的に整数倍である」とみなします。

これにより、ユーザーが正しく `0.005` と入力したのに、コンピュータの誤差でエラーになってしまう理不尽を防いでいます。

## 4. Zod ライブラリの活用

バリデーションの実装には **Zod** というライブラリを使用しています。
TypeScriptと非常に相性が良く、型定義とバリデーションルールを同時に書けるのが特徴です。

### 主な機能
- **`z.coerce.number()`**: 文字列として入力された数字（例: `"100"`）を、自動的に数値の `100` に変換します。
- **`refine()`**: 「0.0025の倍数か？」のような、標準機能にない独自のチェックルールを追加するために使います。
- **`transform()`**: 入力値を加工します。ここでは空文字 `""` を `NaN` に変換して、後のチェックで弾けるようにしています。

### コードの読み方例

```typescript
export const setupFormSchema = z.object({
    laserPower: z.coerce
        .number()                        // 1. 数値に変換
        .min(0, "0以上の値を...")         // 2. 最小値チェック
        .max(1000, "値が大きすぎます"),    // 3. 最大値チェック

    fiberX: z.coerce
        .number()
        .int("整数で入力してください"),    // 4. 整数チェック（小数不可）
});
```

このように、メソッドチェーン（`.` でつなぐ書き方）で直感的にルールを記述しています。

## 5. まとめ

- **仕様**: 入力値はハードウェアの制約（`0.0025`ステップ）に従う必要があります。
- **技術**: 浮動小数点の誤差を考慮し、`EPSILON` を用いた判定ロジックを実装しています。
- **実装**: Zod を使用して、型安全かつ宣言的にバリデーションを定義しています。
- **設計思想**: フロントエンドでは「厳格なチェック」を行い、バックエンドでは「安全な変換（丸め）」を行うことで、システム全体の整合性を保っています。
